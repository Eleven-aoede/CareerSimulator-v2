# Career Simulator

## Goal

- 产品总方向是“职业模拟 + 就业指导”。
- 第一阶段先做“职业模拟”。
- 用户提供岗位名称、岗位 JD、公司行业、公司规模后，AI 生成一个文本支线选择故事，让用户体验该岗位的大致工作情境、压力来源、成就时刻、角色感受，以及这个岗位最常见任务的一次 mini 版本。
- 用户在故事中做出选择，系统根据选择推进事件、调整体验，并最终导向不同结局。

## MVP Definition

- 先不做完整就业指导，只做职业模拟核心链路。
- 核心交付物是一个可游玩的固定交互文本，可直接嵌入 HTML 文件运行。
- 模拟内容要同时包含：
  - 稳定背景块：岗位环境、团队关系、汇报结构、日常节奏
  - 事件块：任务冲突、压力事件、成就事件、协作事件、风险事件
  - 典型任务试做块：呈现该岗位最常见任务的一个具体 mini 版本，记录用户当下情绪和主观难度感
  - 结局：至少能体现“适配度/状态/结果”的差异，并包含轻量匹配度分析与原因说明

## Inputs

- 岗位名称
- 岗位 JD
- 公司行业
- 公司规模
- 用户工作偏好与行为画像

## Output

- 一个文本支线选择故事
- 一份可被 HTML 直接读取的 JSON 模拟脚本
- 故事中包含多个选择点
- 选择会影响后续事件和最终结局
- 用户能够从故事中感受到该岗位的大致工作状态与主观体验
- 用户画像会影响更自然的选项、心理活动、压力来源和成就来源
- 故事中加入一次“该岗位最常见任务”的 mini 模拟，直接给出一小段原始任务素材，让用户真的处理一小步，再选择面对该任务时的情绪，并对体感难度做五点评分
- 该 mini 模拟默认不改变后续主分支，只作为最终结局描述的参考信号
- 用户画像的主路径改为前端生成结构化 JSON，开放式对话采集只作为缺失时的兜底方案
- 前端生成的画像输入需要和岗位信息一起组成统一输入载荷，再交给故事生成器
- 如果进入兜底对话采集，AI 形象仍然偏朋友式，语气温和亲近
- 兜底对话采集只固定第一问；从第二问开始，AI 需要根据用户回答动态回应并调整追问
- AI 引导者设定固定为“小可”
- 整个体验包装为一场“职业穿越游戏”，小可是固定带领者
- 小可的开场白与进入画像采集前的承接语已固定，后续所有用户可见话术都应沿用这套定位，不再来回漂移
- 故事表达需要比纯说明文更有画面感和现场感，但仍以岗位真实性为前提
- 默认按“第一次进入这个行业”的用户来写，少用黑话和默认知识
- 每个场景优先拆成 2-3 个小步骤的连续选择，而不是一步跨太大
- 已拆分出 `references/story-node-format.md` 和 `references/option-generation.md` 作为可复用生成参考

## Scope

- In scope:
- 基于岗位信息生成职业体验型互动文本
- 在模拟前收集用户画像并生成个体化版本
- 用事件和选择模拟工作中的典型情境
- 输出多个描述性结局，并附带体验层面的匹配度分析

- Out of scope:
- 暂不做真实招聘匹配
- 暂不做简历诊断
- 暂不做岗位推荐系统
- 暂不做带数据证据的就业结论
- 暂不做复杂数值化职业测评

## Open Questions

- 用户扮演“新入职员工”还是“候选人入职后的第一阶段”
- 模拟时长目标是多少
- 一个故事需要多少个分支节点才算够用
- 用户画像题目要固定几题，还是动态追问

## Constraints

- 第一版优先验证体验价值，不追求复杂产品形态
- 模拟内容应当让用户感觉“像真的在上班”，而不是泛泛的职场鸡汤
- 需要兼顾可玩性和岗位真实性，但不承诺完全还原真实公司环境
- 结局以描述性反馈为主，可以有轻量匹配度分析，但不采用裁判式评价
- 叙事不能过于流水账，需要有更生动的语言和场景感
- 选择不能默认用户懂行业黑话或专业缩写
- 行动步骤要细，优先写生活化的小动作和小判断

## Decisions

- 产品总方向先定为“职业模拟 + 就业指导”
- 当前阶段聚焦“职业模拟”
- 当前交互形态定为“文本分支选择故事”
- 最终落地形态调整为“固定交互文本 + HTML 承载运行”，而不是持续依赖开放式对话生成
- skill 最终输出格式收敛为 JSON，用户通过上传 JSON 到 HTML 页面来运行模拟故事
- 当前输入字段定为：岗位名称 + 岗位 JD + 公司行业 + 公司规模
- 模拟开始前加入用户画像收集，但主实现改为前端生成结构化 `user_profile`
- 用户画像会影响选项倾向、行为模式、言语模式和心理活动
- 结局改为描述性而非评价性
- 用户画像提问默认使用开放题，必要时才给少量动态选项
- 用户画像阶段的语气定位为朋友式陪聊，而非测评师
- AI 人设固定为“小可”，作为职业模拟支线的引导者
- 小可对用户说话时要始终像在带人进入一场职业穿越体验，避免跳出体验层解释后台机制
- 面向用户的文案里不要出现 `skill`、`json`、`脚本`、`HTML` 这类制作层词汇
- 固定产品级导语可放在网页层，不强制写入 skill 输出
- 小可的语气定位为欢快温和、会回应确认、语言灵动
- 故事文风要在保留真实性的前提下提升画面感，避免流水账式叙述
- 故事分支要细化成更短、更白话的连续选择
- 节点和选项规则单独维护为 reference，不再完全塞在 skill 中
- 在主故事中加入一次“最常见任务 mini 模拟”，记录用户的情绪选择和五点评分难度
- 该 mini 模拟不改动后续主分支，但要进入结局总结
- 结局可加入轻量匹配度分析，说明用户与岗位工作方式更贴合或更拉扯的原因
