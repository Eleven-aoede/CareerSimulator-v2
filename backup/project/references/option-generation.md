# 选项与数值生成规则 v2

## 1. 目的

让交互里的每个选项都像真实工作现场里会做出的一个小动作，而不是测试题答案。

同时保证：

- 选项读起来够白话
- 选项之间平行
- 选项能推动下一节点
- `effect.fit / stress / growth` 的变化有一致逻辑

## 2. 选项总原则

- 每题默认 2-3 个选项。
- 每个选项只表达一个动作、一个处理方式，或一种即时感受。
- 同一题里的选项要平行，不能一个很具体、一个很抽象。
- 选项长度尽量控制在一眼能读完的范围。
- 用户第一次接触这个行业也应能看懂。

不要写成：

- `开展桌研`
- `沉淀洞察`
- `同步框架`
- `打通链路`
- `协同 owner 推进`

如果必须出现专业内容，也要翻译成用户能立刻接住的动作。

## 3. 不同节点该用什么类型的选项

### 3.1 `intro`

固定只有一个按钮：

- `进入这段职业模拟`

不要在 `intro` 给复杂分支。

### 3.2 `node1`

优先类型：

- 状态型
- 起手动作型

例如：

- 干劲一下被拉起来了
- 先稳一下，再看清楚
- 先把材料顺着看一遍
- 先记下最想确认的地方

### 3.3 `node2`

优先类型：

- 处理不清楚信息的动作型

例如：

- 先自己查一下
- 先把问题记下来
- 先找人确认一下边界

### 3.4 `node3`

优先类型：

- 对外连接动作型

例如：

- 先把问题整理好再发出去
- 先顺着对方的方向继续往下做
- 先找同事过一眼

### 3.5 `taskAction`

优先类型：

- 处理素材的第一步

例如：

- 先抓一句最像重点的话
- 先按主题分成两三类
- 先写一句最短总结

### 3.6 `taskEmotion`

优先类型：

- 即时感受记录型

例如：

- 有点上头，想把它理顺
- 有点紧，怕自己抓不到重点
- 有点抗拒，这种活让我先泄气

### 3.7 `taskDifficulty`

固定类型：

- 1-5 五点评分

例如：

- 1 分，挺顺手，知道从哪儿下手
- 2 分，有点绕，但能慢慢理出来
- 3 分，需要认真顶着做
- 4 分，明显开始有压力
- 5 分，光开始就有点困难

### 3.8 `node4`

优先类型：

- 面对压力的即时应对动作

例如：

- 先把顺序排出来
- 先确认最急的是哪一部分
- 先顶着做一版再说

### 3.9 `node5`

优先类型：

- 对当下体验的最后一个收口动作或判断

例如：

- 先把这口气放下来
- 先把结果交出去
- 先确认自己下一次会怎么做

## 4. 选项措辞要求

优先写法：

- 先……
- 有点……
- 想……
- 把……理一下
- 找……确认一下

避免写法：

- 制定策略
- 统筹推进
- 输出结论
- 拉齐认知
- 完成闭环

判断标准：

- 像人在工位上会说的话
- 不是写给领导的周报语言

## 5. `effect` 数值规则

每个选项都必须有：

```json
{
  "effect": {
    "fit": 0,
    "stress": 0,
    "growth": 0
  }
}
```

### 5.1 取值范围

建议只使用：

- `-1`
- `0`
- `1`
- `2`

不要使用过大的跳变值。

### 5.2 `fit`

表示这个选择和岗位要求、以及用户的自然工作方式有多贴合。

加分常见情况：

- 处理方式和岗位日常要求贴合
- 选择体现出适合这份工作的自然姿态
- 在关键节点接住了岗位最核心的要求

减分常见情况：

- 明显回避这份岗位的核心日常
- 处理方式和岗位节奏长期冲突

### 5.3 `stress`

表示这个选择会不会让压力更往上顶。

加分常见情况：

- 硬顶模糊任务
- 在高压下继续自我施压
- 明明需要确认却一直不确认

减分常见情况：

- 做了一个能稳住节奏的动作
- 选择帮助自己先把边界弄清楚

注意：

- `stress` 高不等于 `fit` 一定低
- 有些用户会“很适合做这件事，但做起来也很累”

### 5.4 `growth`

表示这个选择有没有推动用户更进入这份工作的学习或适应过程。

加分常见情况：

- 主动处理信息
- 做出一次有效确认
- 愿意进入岗位核心任务

减分使用要谨慎，通常不必频繁给负值。

## 6. 数值分配的整体逻辑

### 6.1 不要把“最乖的选项”永远写成唯一高分

选项之间应该体现风格差异，而不是道德高低。

例如：

- “先确认清楚”可能 `fit +1 / stress -1`
- “先自己试一版”可能 `fit +1 / stress +1 / growth +1`

两者都可以成立，只是体验不同。

### 6.2 典型任务模块要更清楚地拉开差异

在 `taskAction / taskEmotion / taskDifficulty` 中：

- 用户对核心任务更顺手，应提高 `fit`
- 用户明显更紧或更抗拒，应提高 `stress`
- 愿意进入材料本身，应提高 `growth`

### 6.3 `taskDifficulty` 的极端分值要影响结局

建议：

- 1-2 分：`fit` 可偏高，`stress` 可偏低
- 3 分：中性
- 4-5 分：`stress` 应明显提高，`fit` 可适度下降

## 7. 分支一致性

写选项时要同时检查：

- 这个 `next` 合不合理
- 下一节点能不能接住这个动作
- 数值变化和文案是否一致

例如：

- 用户选了“先找人确认一下边界”
- 下一节点就不该写成“你决定一个人闷头做完”

## 8. 用户画像怎么影响选项

如果 `profile_skipped = false`，选项措辞和数值应轻微向用户倾向靠近。

例如：

- 用户偏“先整理再确认”，就可以多给这一类自然选项
- 用户对高压敏感，硬顶类选项的 `stress` 可以更高
- 用户重视推进和产出，推进类选项可适度提高 `fit` 或 `growth`

如果 `profile_skipped = true`：

- 不要写得像“已经知道这个用户是谁”
- 选项保持中性、均衡、普适

## 9. 自检问题

生成完一组选项后，至少检查：

1. 这些话像人会说的吗？
2. 这些选项是在同一个层级上吗？
3. 选项是不是一个小动作，而不是一整套策略？
4. 数值变化和文案是不是一致？
5. 选完以后，下一节点能接得上吗？
