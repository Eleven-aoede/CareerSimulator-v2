# 项目优化路线

## 待优化项

### 1. 流程管理 Agent

**现状**：当前阶段转换依赖 LLM 在回复末尾输出 `<extraction>` 标签，由 prompt 指令约束。实际运行中 LLM 可能提前/遗漏输出该标签，导致阶段转换不稳定。

**方案**：考虑新增一个独立的 flow-control agent，专门负责判断当前对话是否满足进入下一阶段的条件，将"内容生成"与"流程决策"解耦。

**待定事项**：

- 需要多次端到端测试，定位当前方案的具体失败场景（提前转换 / 遗漏转换 / 信息不完整就转换）
- 评估额外 agent 调用带来的延迟和成本
- 确定 flow-control agent 的判断输入（完整对话历史 or 最近 N 轮）

---

### 2. 管理员模式

**需求**：在前端输入特定校验码后进入管理员后台，可查看所有用户的对话日志和系统运行状态。

**设计要点**：

- 前端：登录页增加校验码入口，通过后切换至管理员视图
- 后端：新增管理员鉴权接口，校验码存储在环境变量中
- 管理员后台功能：
  - 查看所有用户列表及其当前阶段
  - 查看任意用户的完整对话历史
  - 查看 LLM 请求/响应日志
  - 查看系统错误日志

---

### 3. 进入剧情模拟需点击两次

**现象**：画像收集完成后，第一次点击"开始剧情"按钮会提示"尚未完成信息收集"，第二次点击才能正常进入。

**可能原因**：前端阶段已切换到 `story_simulation`，但后端 state 尚未持久化（竞态），或前端在首次点击时 `job_input` / `user_profile` 数据还未同步到 state。

**待排查**：
- 确认是前端 state 未就绪还是后端接口校验未通过
- 检查 `startStory()` 时后端返回的错误响应内容
- 可能需要在阶段切换完成后强制等待一次 `refreshState` 成功再启用按钮

---

### 4. 匹配度 / 压力 / 成长评分标准

**现状**：当前三维评分（fit / stress / growth）由生成剧情节点的同一个 LLM 调用自行打分（在 `option-generation.md` prompt 中给出 rubric），打分和内容生成耦合在一起，可能导致：
- 分数波动大，同一场景不同运行结果差异明显
- LLM 同时承担"写故事"和"评估匹配度"两个目标，注意力分散
- 缺乏与用户画像的精确关联（当前仅在 prompt 中要求"轻微靠近"）

**方案**：考虑新增独立的 scoring agent，将打分与内容生成解耦：
- 内容生成 agent 只负责生成节点文案和选项描述
- scoring agent 在用户做出选择后，结合用户画像、岗位信息和选择内容，独立给出 fit / stress / growth 分数
- scoring agent 可以使用更小/更快的模型，降低成本

**待定事项**：
- 确定 scoring agent 的输入格式（节点上下文 + 选择 + 用户画像）
- 是否需要在 scoring agent 的 prompt 中硬编码更严格的评分量表
- 评估独立打分调用带来的额外延迟（是否可以和下一节点生成并行）
- 最终结局的 bucket 划分阈值是否需要调整（当前：fit>=9 high, fit>=3 mid, else low）

---

### 5. 节点选择存储与回退

**需求**：存储用户在每个节点做出的选择，支持回退到之前的节点重新选择。

**设计要点**：
- 后端：`story_state.choices` 已记录选择历史，需新增回退接口（删除目标节点之后的所有选择和生成数据，回退分数）
- 前端：在已完成的节点卡片上显示"重新选择"入口
- 注意事项：
  - 回退后需要重新生成后续节点（因为剧情依赖前序选择）
  - 需要清除回退点之后的 `generated_nodes` 和 `choices`
  - 分数需要从累计值中减去被回退节点的增量
